  • tsc：tsx的编译器

  • rollup：对于工具库、组件库来做打包构建友好，简单，早期提出treeshaking概念

  • tsup

  • vite

1、tsc相关

1.1、tsc（TypeScript编译器）

用来编译TS的工具，随着rust、go语言出来，使用JS去写编译器像Babel，那么它在性能上、内存安全上有问题，将ts代码编译成js，在tsconfig.json上进行配置

特点：

  • 自家的，专注于ts编译，涉及样式的，使用不了（post css）js编译， 类型输出 （dts）
  • 强类型的检查

缺点：

  • 仅仅支持TS编译，
  • 无法代码拆分，模块打包
  • 性能差，大规模项目性能堪忧
  • 配置繁琐，tsconfig难配置， 有时需求借助babel，postcss(postcss.config.js)

在构建的时候，都能形成一套AST，如果AST不能进行复用的话会造成极大的浪费

1.2、使用场景

  • 早期项目，没有复杂的编译需求
  • 工具类，后端 node （没有样式、字体等打包需求）

2、rollup

随着项目复杂度提高，tsc满足不了要求，这个时候要求有一个开箱即用的方案接近于0配置，rollup成为工具库首选构建打包方案

2.1、特点

  • tree-shaking：移除未使用代码
  • 简化配置：tsc、webpack
  • 丰富的插件生态,将功能的生态构建交给全世界的开发者，框架插件规则制定者

2.2、适应场景

  • UI库
    1. mantine（国外）
  • 工具库：react-use
  • 按需导入的场景

2.3、缺点

  • 大型业务项目不适合，vite、webpack适合
  • 热更新方面
  • 性能方面

可以构建出多场景的包：浏览器端-ESM方式；node：commoljs方式；或许还有其他的，如AMD，CMD，common GS UMD等，output部分支持数组，通过数组在这里指定format’就可以打出不同的产物

多产物模块化的输出

压缩：rollup-plugin-terser

3、tsup

基于go语言，在ES build的基础上，go开发了ES build，ES build在它的基础上上层有了tsup, 直接使用ESbuild，会有许多需要我们自己处理如样式内容处理，还有类型文件输出；tsup在ES build的基础上做了一层封装，使用更简洁

3.1、特点

  • 0配置，默认支持ts
  • 性能非常好
  • 多格式输出
  • 代码的动态导入，代码分割

3.2、使用场景

  • 工具库
  • 脚手架

3.3、缺点

  • 业务构建不推荐
  • 插件生态没有rollup 繁荣

整体分为2部分：JS的构建（几毫秒）；类型的文件（Dist）的构建

4、vite

整合esbuild、rollup的现代构建工具

4.1、特点

  • 开发环境基于esbuild
  • 按需加载
  • 热更新 HMR
  • 生产构建 rollup

4.2、适应场景

  • 大型业务场景
  • Vue React业务项目

4.3、缺点：

  • 生态
  • 低版本兼容，虽然有lahecy兼容方案，但是调试需注意

二、不同形态前端项目打包构建有什么具体的设计思路和实践经验

1、工具库

工具库是最轻量级的前端项目形态，如日期处理、算法、HTTP请求工具等，这类项目的特点是体积小、依赖少、通常面向Node.js或浏览器环境提高支持，不涉及样式、字体、静态资源

设计思路：

  1. 支持多模块格式：
    • 输出多种模块格式（CommonJS、ESM、UMD），以便进入不同的消费场景
  2. 启用tree-shaking
    • 保证消费者可以按需引入功能模块，减少无用代码
  3. 类型支持：
    • 提供完整的TypeScript类型声明（d.ts文件）

实践经验：

  • 工具选择：
    • 使用Tsup:极速构建、零配置即可支持多模块输出和类型声明
    • 使用Rollup:当需要更细粒度的控制输出是（如手动指定模块拆分）

2、组件库

组件库是面向UI的开发库，通常为React、Vue或者其他框架提供通用组件，特点是可能包含多个独立模块，依赖框架环境，并需要支持按需加载

设计思路：

  1. 支持按需加载
    • 输出每个组件的独立文件，避免消费者加载多余的代码
  2. 打包样式：
    • 将CSS打包到独立文件中
  3. 类型声明：
    • 提供TypeScript类型定义，特别是对组件props的类型支持

实践经验：

  • 工具选择

    Rollup：更适合组件库，因为可以灵活的定义多个输出目标

    Tsup：适合简单组件库的快速开发和打包

  • 按需加载：配合babel-plugin-import（用于React）或类似工具，在构建时输出多个组件模块

分包技巧：利用 preserveModules: true 配置。当组件库组件较多，开启该配置后，Rollup 会把每个模块单独输出为一个文件，而非合并成一个大文件 ，输出结构会和输入的模块结构对应，方便按需加载、调试和维护，输出到 dir 指定的目录（如示例里的 dist ），格式设为 esm 利于现代构建工具（Vite 等）处理。

作用：按需加载、模块隔离与测试、依赖清晰（后续迭代、维护时降低误改风险）

3、单页应用（SPA）

最常见的前端项目形态，框架如React、Vue或者Angular，特点开发时需要快速迭代与热更新，生产时需要优化资源加载

设计思路：

  1. 开发体验优先：
    • 启用快速热模块替换（HMR）和增量编译
  2. 优化生产构建
    • 对静态资源（JS\CSS\图片等）进行拆分和压缩
  3. 按需加载和代码分割
    • 使用动态导入来实现按需加载模块

vite，webpack

面试回答；：

在实际开发中，我接触过工具库、组件库以及单页应用（SPA）三种主要形态的前端项目。不同形态对应不同的构建目标，我会根据项目特点选用合适的构建工具和策略，兼顾开发效率和产物质量。以下是我在这三类项目中积累的打包构建设计思路和具体实践经验。

1、工具库

工具库是最轻量级的项目，一般不涉及样式、资源，目标是给其他项目提供纯逻辑功能，比如加解密、格式化、请求封装等等。

在这类项目中，我的构建设计主要有三点：

  • 第一，支持多种模块格式输出，比如 CommonJS、ESM 和 UMD，以适配 Node.js、浏览器或打包工具等不同环境；
  • 第二，支持 tree-shaking，为了让使用者按需引入功能，避免不必要的代码；
  • 第三，提供完善的 TypeScript 类型声明，让用户在使用时获得良好的智能提示和类型约束。

在工具方面，我首选的是 **Tsup**，它非常适合工具库构建，配置简单，还能自动输出多格式和类型文件。对于需要更细致控制的情况，我会使用 Rollup，比如需要手动拆分模块或插入一些特定插件时

2、组件库

组件库开发相比工具库更复杂一些，不仅要输出组件代码，还要处理样式、类型、模块结构等问题。为了让组件库更易用、更灵活，我通常遵循以下几个设计思路：

  • 首先，支持按需加载，每个组件都会单独输出为一个模块；
  • 其次，样式要独立打包，避免样式污染或冲突；
  • 最后，类型定义要完整，特别是组件的 props 类型，这对使用者来说非常重要。

在实践中，我更倾向使用 Rollup 来打包组件库，因为它更灵活、插件多，适合复杂的输出控制。同时，我也会配合使用 preserveModules: true 这个配置，让每个组件保持独立模块输出，便于后期维护和按需加载。

如果项目对开发效率要求高，也可以选择 **Tsup** 来快速构建基础版本，然后再通过插件或辅助配置做样式处理。

3、SPA

SPA 是最常见的前端项目形态，构建目标分为两个方面：开发阶段的快速迭代 和 生产阶段的性能优化。

开发阶段我注重的是开发体验，比如启用 **热更新（HMR）**、支持 增量构建，能显著提高效率。

而在构建上线前，我会重点关注几个优化点：

  • 第一，代码分割和按需加载，通过 import() 动态导入来实现；
  • 第二，资源压缩，包括 JS、CSS、图片等都需要压缩和懒加载；
  • 第三，构建速度和插件生态，我优先选择 Vite，因为其原生支持 ESM 和更快的冷启动体验；对于老项目或者需要复杂自定义的情况，仍会使用 Webpack。

Vite 的插件生态也很成熟，比如自动压缩、CDN 注入、PWA 等插件，都可以大大简化配置。

整体来说，我会根据项目类型灵活选择构建策略：工具库追求轻量和兼容性，用 Tsup 快速搞定；组件库注重模块拆分和样式隔离，用 Rollup 更易控制结构；SPA 项目优先开发体验和性能优化，Vite 是我目前的首选方案。

三、如果让你设计一个打包工具，你怎么设计

设计目标：

  1. 快速开发启动：利用ES模块加载，尽量减少服务器启动时间
  2. HMR（热模块替换）：支持快速反映文件更改，无需重新加载整个页面
  3. 插件化架构：允许通过插件扩展功能，提高工具的灵活性和可扩展性
  4. 生产构建优化：使用像Rollup这样的高级打包工具进行代码拆分，压缩和优化
  5. 编译侧能力：编译工具的sorcemap、tree-shaking、chunk机制
  6. 运行时优化：模块缓存，hmr等

核心架构组件

  1. 开发服务器

    开发服务器应处理Http请求并根据需要提供ES模块，使用Node.js的http模块可以快速启动基本服务器，同时处理文件的即时转译

  2. 插件系统

    设计一个插件系统，使得功能如预处理，HMR，压缩等可以轻易地添加或修改

如果让我设计一个前端打包工具，我会聚焦在开发体验和生产优化两个方面。开发阶段，我会利用浏览器原生 ES 模块加载来实现快速启动，并引入 WebSocket 支持的 HMR（热模块替换），让文件更改能即时反馈到页面上。整体架构上我会设计一个 插件系统，参考 Vite 和 Rollup 的钩子机制，支持如 transform、resolveId 等阶段扩展，方便用户定制行为。构建阶段，我会集成 Rollup 作为底层引擎，实现 Tree-shaking、代码分割、资源压缩等高级优化，同时生成 sourcemap 和依赖可视化报告。在运行时，我也会引入模块缓存机制，加速模块热替换和页面加载。最终，这个工具将支持多目标输出，比如工具库、组件库或 SPA 应用，并通过 CLI 和插件系统做到轻量、灵活和高性能。

🟦 一、设计目标讲解

“在目标设定方面，我会关注以下几个核心点：”

  1. 快速开发启动：利用浏览器原生 ES 模块（ESM）加载，避免传统打包过程，启动即服务。
  2. 热模块替换（HMR）：实现文件修改后只替换变动模块，提升前端开发迭代效率。
  3. 插件化架构：提供类 Vite 的插件机制，方便用户按需扩展功能，如压缩、Mock、国际化等。
  4. 生产构建优化：集成 Rollup 作为打包核心，支持代码分割、Tree-shaking、资源压缩。
  5. 编译侧能力：包括生成 SourceMap、静态依赖图分析、自动 chunk 拆分。
  6. 运行时性能优化：模块缓存、懒加载、预构建机制，提高实际加载性能。

🟨 二、核心模块架构

“在架构设计上，我会将打包工具划分为以下几个核心模块。”

1. 开发服务器（Dev Server）

  • 直接使用 Node.js 构建 HTTP Server；
  • 请求时按需读取文件内容，识别 .ts, .jsx, .vue 等后缀；
  • 实时进行按需转译（比如通过 ESBuild 编译 TS、JSX）；
  • 支持 HTML 入口解析、资源路径重写、模块预加载等功能。

2. 插件系统（Plugin System）

  • 插件机制是整个工具的核心扩展点；
  • 我会设计类似 Rollup/Vite 的钩子函数，比如：
    • resolveId、transform、load、generateBundle 等；
  • 插件可覆盖构建各阶段，比如压缩、预处理、Mock、国际化替换等；
  • 支持用户开发插件，同时内置常用插件（如 alias、css、hmr）。

🗣️ 面试话术小贴士：“我设计插件系统的核心理念是开放性，用户可以用插件改变任何构建行为，甚至替换默认的 loader。”

🟥 三、HMR（热更新机制）

“HMR 是现代开发体验的重要一环。我的设计中会：”

  • 利用 WebSocket 建立客户端与服务端连接；
  • 文件变动后，构建模块依赖图，找到变动模块及其影响范围；
  • 向浏览器推送模块热替换消息，客户端接受后只替换相关模块；
  • 对 CSS / Vue / React 等框架提供定制化更新策略。

🟧 四、生产构建阶段

“在生产构建方面，我会使用 Rollup 作为底层打包引擎，因为它更擅长处理模块树和资源优化。”

  • 通过配置实现 Tree-shaking、代码压缩、资源 hash 命名；
  • 配合动态导入语法进行 chunk 拆分；
  • 输出多种格式（如 ESM + CJS + IIFE）适配不同运行环境；
  • 支持静态分析输出报告（比如各模块体积、依赖关系图）

问题1：你了解 pnpm 吗？它比 npm / yarn 有哪些优势？

它最大的优势之一是使用了软链接机制：在安装依赖时，不再将依赖复制到每个项目中，而是将它们统一保存在一个中央存储区中，通过软链接（就像 Windows 中的快捷方式）来指向这些真实依赖。这种方式极大减少了磁盘占用，同时提升了模块复用率。此外，pnpm 还有很好的缓存机制和并行安装能力，能有效提升安装速度。它还原生支持 workspace，在管理多个子项目时比 npm/yarn 更加高效。整体来看，pnpm 是一个以‘中心化依赖复用’为核心思想、兼顾性能和磁盘效率的现代包管理工具，非常适合工程规模较大的场景。

相比 npm 的扁平结构和 yarn 的 node_modules 链接方式，pnpm 会严格按照依赖声明生成树状结构，能更早发现重复或错误依赖。

在组件库开发中，我使用了 pnpm 来搭建 monorepo 结构。pnpm 原生支持 workspace，我们把每个组件作为独立的子包管理，同时通过软链接机制在本地模拟依赖关系，开发时修改某个子包能立即生效，极大提高了联动调试效率。相比 npm 或 yarn，pnpm 的依赖安装速度更快、磁盘占用更小，并且强依赖隔离能防止组件间的幽灵依赖和版本冲突问题。整体体验非常适合做组件库这类多模块、多依赖、迭代频繁的工程。

快速安装与缓存

  • pnpm 的安装非常快，支持并行下载；
  • 所有安装包统一缓存，无需多次重复拉取；
  • 对于频繁运行的构建脚本（如 Storybook/Vite 打包），速度有明显优势。

依赖隔离与版本控制

  • pnpm 遵循严格的依赖隔离机制，不允许未声明的依赖被隐式使用；
  • 每个包需要明确声明依赖，方便组件库后续维护、升级和发布；
  • 可配合 pnpm recursive publish 实现按包独立版本管理或统一发布。

React的事件机制

一、是什么

React基于浏览器的事件机制自身实现了一套事件机制，包括事件注册、事件注册、事件冒泡、事件派发。在react中，我，们称为合成事件，合成事件就是react模拟原生DOM事件所有能力的一个事件对象。可以这么理解：原生浏览器事件的跨浏览器包装器

区别：

  1. 事件名称命名：

    • `JavaScript``

      <button onclick="handleClick()">按钮</button>

    • React

      const button = <button onClick={handleClick}>按钮</button>

    react通过驼峰的命名方式，原生是小写的方式，事件绑定的函数不同：react：传入对应的函数；原生：字符串

  2. 事件处理函数不同：

    React传入一个回调函数，原生传入一个字符串

一、React首页白屏可能是什么问题引起的？怎么解决

React性能优化技巧：解决首页白屏加载问题，提升用户体验 - 云原生实践

白屏主要发生在首次加载时，主要原因包括

  1. 静态资源路径错误。
  2. 依赖包没有正确安装或版本兼容问题
  3. 网络请求失败
  4. react的路由配置错误
  5. 构建工具的问题，比如 Webpack 配置错误。
  6. 资源加载阻塞：React应用的JavaScript文件较大，加载和解析时间较长，导致浏览器在渲染页面之前需要等待。
  7. 同步渲染：React默认的渲染方式是同步的，即在所有JavaScript代码执行完毕后才会进行DOM渲染。
  8. 服务端渲染（SSR）缺失：单页应用（SPA）通常依赖客户端渲染，服务端返回的初始HTML内容较少，导致浏览器渲染延迟。

  • 解决办法：

    1. 确保静态资源的路径配置正确，检查 React项目的 config文件或者 Webpack 的配置文件。

    2. 确认所有依赖包已经正确安装，版本兼容。如果有多个版本的 React 依赖包如react router/redux，可能会导致冲突

    3. 确保网络请求的 API正常工作，可以通过 console日志或chrom devtools 验证

    4. 检查 ReactRouter 的配置，包括路径匹配和重定向设置。

    5. 审查Webpack配置，特别是 output和 publicPath 设置

    6. 使用服务端渲染（SSR）

      使用next.js等SSR框架，在服务器上渲染React组件，生成完整的HTML，然后发送到客户端

    7. 预渲染：是一种介于客户端渲染和服务端渲染之间的技术，适用于静态内容的页面。

      • 使用prerender-spa-plugin等工具。

      • 在构建阶段生成静态HTML文件

      • 在用户访问时直接返回预渲染的HTML。

      注意：刷新时有时会闪烁以下首次页面，这时解决方法，我们可以设置两个预渲染页面（a.html-首页；b.html-loading图）这时候，如果加载首页，直接a.html；如果加载其他的页码并没有设置预渲染页面，就用b.html，这时候就需要用nginx配置

    8. 懒加载与代码拆分

      可以将应用拆分成多个小的代码块，按需加载，减少初始加载时间。

      • 使用React.lazy和Suspense实现组件级别的懒加载。
      • 使用webpack的code splitting功能进行代码拆分。

    9. 优化资源加载

      通过压缩代码、优化图片、使用CDN等方式，可以减少资源加载时间。

      • 使用UglifyJS、Terser等工具压缩JavaScript代码。
      • 使用ImageOptim等工具压缩图片。
      • 使用CDN加速资源加载

    10. 添加启动屏，缓解用户对白屏的感知

二、前端如何处理后端接口一次性返回的超大树形结构数据

对于前端来讲，关键在于减少渲染压力、提高用户的交互体验，一般考虑以下几个方面：

  1. 分页加载：将数据页分成多页，用户滚动或者点击“加载更多“的时候加载新一页的数据

  2. 虚拟滚动：只在视口内渲染一定数量的节点，随着用户滚动动态加载更多的节点。

    树形组件库：React的react-virtualized

  3. 懒加载：按需加载数据，只有在用户展开某个节点时才请求该节点的子节点数据

    除了懒加载数据，惰性渲染也是一种优化手段，即只在需要的时候才渲染DOM节点。比如， 当用户展开树节点时才渲染其子节点

  4. 批量处理：将大数据集分块处理，避免一次性渲染大量数据，造成浏览器卡顿。

  5. Web Workers：当进行复杂的计算或处理时使用Web Workers，可 以将计算从主线程分离出来，避免主线程阻塞，提高性能。

  6. 使用缓存：对于大型树形数据，可以考虑使用本地缓存(如localstorage或IndexedDB)来存储已经加载过 的数据，避免每次用户操作时都去请求后端

  7. 树形结构优化:优化树形结构数据本身，例如对于深度很深但不常用的节点，考虑在后端进行预处理筛选， 或者返回数据的层次结构时，先返回主要的部分，再按照需求加载剩余的部分
